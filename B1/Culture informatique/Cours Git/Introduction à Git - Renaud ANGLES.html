    Introduction à Git¶

    A vous de jouer

    Qu'est ce que Git ?

    Avant propos¶

    Disclaimer

    Ce cours n'a pas vocation à apprendre à maitriser Git. Il est même possible de considérer que l'usage qui en est présenté ici est dévoyé, car la présentation qui en est faite à vocation à vous fournir un socle de connaissance suffisant pour que vous puissiez télécharger les codes sources qui seront utilisés par les intervenants tout au long de votre cursus.

    Par exemple, nous ne verrons pas ici comment configurer git de manière à créer un projet côté client puis à le déposer sur un serveur.

    Ce que nous verrons ici c'est :

    • comment interagir avec un plateforme Git simplifiée (Github ici, mais pourquoi pas Bitbucket ou Gitlab)
    • Commment créer un gestionnaire de version local afin de ne jamais perdre vos différentes versions et de les sauvegarder
    • Comment fonctionnent les markdowns

    Tout part d'un besoin¶

    Git, SVN, Mercurial ?! Qu'est ce donc et pourquoi a-t-il fallu créer ce genre d'outils surprenants ?!

    A vous de jouer (20 minutes)

    • Installez Git
    • Ouvrez un terminal et tapez la commmande git help
    • Si vous avez une aide sur Git, c'est que vous avez gagné

    Une fois cela fait, vous allez configurer vos identifiants Git (login et mail) de manière à ce qu'ils soient utilisés automatiquement pour tous vos projets :

    1
2
3
4
5
6
7
    # Pour que vos informations soient sauvegardées pour TOUS les projets :
git config --global user.name "Tom SELLECK"
git config --global user.email "tom.selleck@hotmail.com"

# Pour que les informations soient sauvegardées uniquement pour le dépôt en cours :
git config user.name "Tom SELLECK"
git config --global user.email "tom.selleck@hotmail.com"

    Cette opération permet de vous identifier auprès d'un dépôt donné. Nous n'aborderons pas dans ce cours les mécanismes de sécurisation de l'accès par mot de passe ou certificats SSH pour des raisons de temps, de complexité et parce que ce n'est pas vital pour commencer à travailler. Vous trouverez plus de détails sur la sécurisation par mot de passe ici [anglais].

    Récupérer un projet Git existant¶

    Dans toute cette partie du cours, nous travaillons dans le terminal et pas en interface graphique.

    1 - Créer un dossier dédié¶

    Maintenant que Git est installé, nous devons tout simplement créer un dossier dédié au projet et nous y placer :

    1
2
3
4
5
    # Création du dossier
mkdir mon_projet

# Déplacement dans le dossier 
cd mon_projet

    2 - Récupérer le code source :¶

    Une fois cela fait, nous allons rapatrier un projet existant en le clonant :

    1
    git clone https://github.com/Glandalf/starter.git

    Vous pouvez alors constater que vous disposez maintenant d'un dossier starter ; Dossier dans lequel nous allons nous placer :

    1
    cd starter

    Cette étape est importante car il faut être dans ce dossier pour utiliser les commandes Git. Vous devriez d'ailleurs remarques un dossier caché nommé .git dans ce dossier.

    Quel est le code secret ?!

    Dans le cas présent, cela n'a aucun intéret mais vous pouvez à présent modifier le code localement, depuis votre éditeur préféré. De cette manière, vous pouvez d'une part télécharger des codes sources de projets ouverts pour y contribuer ou les utiliser mais vous pouvez aussi télécharger les sources fournies par vos intervenants.

    Gérer vos versions et sauvegardes¶

    1 - Cloner son dépôt Github¶

    La première étape consiste à vous "connecter" sur votre ordinateur au dépôt git distant créé sur Github. Pour se faire, depuis la page principale de votre projet sur Github, récupérez l'URL de votre dépôt :

    1. cliquez sur "Clone or download"
    2. copiez le lien se terminant par .git
    3. clonez le dépôt (comme nous l'avons fait précédemment) dans un dossier dédié :
    1
    git clone https://github.com/YourNickname/youProjectName.git

    Vous voilà revenu au même point que dans l'exercice précédent, mais sur un dépôt dont vous êtes maintenant propriétaire.

    2 - Créer un fichier¶

    Vous allez simplement créer un fichier test.html dans votre dossier et l'ouvrir avec votre éditeur de texte préféré. Ecrivez y quelque chose comme :

    1
2
3
4
5
6
7
    <html>
<head>
</head>
<body>
    <h1>Hello weurld</h1>
</body>
</html>

    Sauvegardez vos changements puis dans votre console, indiquez à Git que vous voulez ajouter vos changements à l'aide de la commande 

    1
    git add .

    L'usage de cette commande permet de référencer tous les fichier de votre projet, créés ou modifiés. Si vous ne faites pas cela, Git n'aura pas "conscience" de vos modifications et ne pourra pas les sauvegarder.

    3 - Enregistrer une version de votre code¶

    Vous devez alors pouvoir enregistrer vos modifications comme une version au sens Git du terme en utilisant la commande commit :

    1
    git commit -m "Création d'un fichier html gentil"

    A partir de ce moment, le client Git de votre ordinateur enregistre l'état actuel de votre code comme une version donnée, que vous pourrez d'ailleurs identifier à l'aide du message que vous aurez spécifié à l'aide du paramètre -m (ici "Création d'un fichier html gentil").

    Un commit peut contenir plusieurs créations/modifications

    Dans l'exemple, pour des raisons de simplicité, nous n'avons créé qu'un fichier mais vous créerez ET modifierez généralement plusieurs fichiers avant de réaliser un commit.

    Vous auriez par exemple pu ajouter un fichier CSS et 2 images, et tout intégrer en un seul commit, exactement comme nous l'avons fait ici. Rien de plus à faire puisque la commande add tient compte de tous les fichiers que vous avez créé et tous ceux que vous avez modifié.

    4 - Gérer ses versions¶

    Nous avons, au sens de Git, deux versions de notre code : la première correspond à la création du dépôt Git avec le fichier readme, la seconde correspond à notre commit.

    Modifions notre code¶

    Nous allons commencer par modifier notre fichier readme.md car nous voulons que sur Github il soit plus sympa à lire.

    A vous de jouer (30 min)

    Prenez un peu plus en main la notation en markdown pour faire une jolie page readme.

    • Ajouter une image sympa (lien vers une image en ligne)
    • Expliquez ce que fait votre code actuellement (page html qui affiche bonjour)
    • Expliquez ce que vous aimeriez que cela devienne
    • Expliquez pourquoi vous avez créé cette page (un gentil prof vous y a contraint).
    • etc.

    Modifier aussi votre fichier HTML pour changer le titre (en écrivant bonjour en Hollandais par exemple)

    Nous avons ici modifié deux fichiers. Les plus téméraires pourront se risquer à créer un deuxième fichier HTML ou un fichier CSS peut-être.

    Récupérer une ancienne version¶

    Si vous vous rendez-compte que vous avez dégradé votre code dans cette dernière version et que vous voulez récupérer la précédente version.

    Identifier le numéro de votre commit à récupérer. Dans l'exemple ci-dessus, nous voulons récupérer le commit "cbb1b476bbba4104db5e8b9599b9b29277c18c0a", mais dans votre cas, ce sera un hash différent.

    Ouvrez un explorateur dans le dossier de votre projet pour constater la magie se produire.

    Dans votre terminal, récupérez la version concernée via la commande :

    1
    git checkout cbb1b476bbba4104db5e8b9599b9b29277c18c0a

    Vous devriez alors voir votre code revenir à sa version précédente.

    Dépôt local vs. dépôt distant

    Depuis que nous avons clôné le dépôt, nous n'avons plus du tout travaillé sur le serveur. Les commits que nous avons réalisé se passent uniquement sur votre ordinateur, et le serveur n'en a donc pas notion.

    Cela n'est pas l'usage le plus fréquent de Git et nous allons devoir le sauvegarder sur le serveur

    Nous allons maintenant revenir sur la dernière version de notre code car c'est bien sur celle-là que nous voulons travailler (il serait dommage de modifier la version antérieure car cela entraine la mise en oeuvre de mécanismes de Git que nous n'aborderons pas dans ce cours pour préserver votre santé mentale).

    A vous de jouer

    Une nouvelle fois, vous allez réaliser un git checkout sur le hash correspondant à votre dernière version.

    Sauvegarder nos changements sur le serveur¶

    Git vs Github

    Nous allons réaliser la sauvegarde de notre code sur le serveur. Dans le cas présent, nous travaillons avec Github mais la manipulation serait exactement la même avec un serveur Git autre, qu'il dispose d'une interface graphique ou non.

    Maintenant que nous sommes revenu à la version à jour de notre code, nous pouvons sortir ce problème de notre tête et nous focaliser sur la sauvegarde sur le serveur à l'aide de la commande push :

    1
    git push -u origin master
    1. Push indique à votre client git que vous voulez enregistrer tous vos commits sur le serveur
    2. l'option -u indique que vous voulez utiliser l'utilisateur que vous avez configuré au tout début de ce cours. Sans cela, votre push sera refusé sur le serveur faute d'authentification. (Etant donné que votre dépôt est public, il ne sera pas nécessaire de spécifier votre mot de passe)
    3. Origin correspond à l'identifiant du dépôt (nous ne détaillerons pas cela ici, nous travaillerons 99% du temps avec origin)
    4. master est le nom de la branche sur laquelle on sauvegarde nos changements. La notion de branches est conceptuellement simple à comprendre mais implique beaucoup de choses que nous n'allons pas évoquer ici car cela fait déjà pas mal d'informations. Pour le moment, nous travaillerons uniquement avec Master.

    Si tout s'est bien passé, vous devriez avoir un message indiquant que tout est sauvegardé sur le serveur.

    Rendez-vous alors sur Github pour vous assurer que c'est bien le cas. Vous trouverez normalement sur la page d'accueil, endessous de la liste de vos fichiers, votre ficheir readme.md mis en forme pour être lisible dans le navigateur.

    Collaborer sur du code¶

    Aller plus loin

    Vous avez peut-être bien compris tous les concepts que nous venons de voir. Cela étant, il ne s'agit que de la partie immergée de l'iceberg. Des outils comme Git sont dès leurs conception pensés pour collaborer entre plusieurs développeurs.

    Dans vos startups par exemple, vous pourriez vouloir travailler à plusieurs sur des fonctionnalités différentes. Et pour faire cela, on va essayer d'oublier le partage de code par Discord avec des copier/coller douteux. Github vous propose un tutoriel pour prendre cela en compte.
